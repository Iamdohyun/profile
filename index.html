표면적인 요구사항 너머의 실제 업무 병목을 파악하는 것을 중요하게 생각합니다. 문제 발생 시 실무 프로세스를 직접 역추적하여 원인을 진단합니다.

식권 정산 오류를 파악하기 위해 6일이 소요되는 수기 정산 프로세스를 처음부터 끝까지 직접 돌렸고, 총무팀 담당자의 하루 업무를 동행하며 어느 단계에서 시간이 허비되는지 확인했습니다. 국내 최초 역발행 승인 프로세스를 도입할 때는 법무팀과 위법 여지를 직접 검토했고, 보수적인 경영지원실 설득에는 타사 사규를 비교 분석한 데이터를 활용해 규제 완화를 이끌어냈습니다.

확인된 문제는 임시방편적 대응을 지양하고, 확장 가능한 정책과 구조로 해결합니다. 모든 의사결정은 데이터(KPI, ROI)를 기반으로 진행합니다. 4년간 이 방식으로 연간 8억 원 비용 절감, 2억 원 신규 매출, 특허 14건을 만들어냈습니다.

01
식권 서비스 — 기업 식권 결제·정산 플랫폼
신규 매출 2억 2024.12 – 2025.07
+
수기 정산, 피크타임 혼잡, 고객사별 파편화된 정책 등으로 운영 확장에 한계가 있던 식권 서비스입니다. 과천 사옥 이전 TF에서 유사 서비스를 분석해 직접 제안한 프로젝트로, 사용부터 PG 결제·정산까지 전 과정을 재설계해 매출 성장과 운영 효율을 한 번에 끌어올렸습니다.
🔴 Problem
식당마다 종이 수기 장부로 운영. 담당자가 엑셀로 타이핑해 정산하면 매번 5번씩 오류가 났습니다. 피크타임엔 주문·계산·적기가 한 카운터에서 뒤엉켜 혼잡이 극심했고, 고객사마다 한도·업종 제한·정산 방식이 달라 신규 도입 때마다 하드코딩으로 임시방편 처리하던 상황 — 도입 고객사는 자사 포함 3곳에 머물러 있었습니다.
🟡 Decision
점진적 개선이 아닌 전면 재설계를 선택. 기존 시스템은 결제 완료 데이터만 다룰 수 있는 구조였기 때문입니다. 정책은 유지보수팀이 큰 고객사 요청만 개별 하드코딩으로 대응하며 기술 부채가 누적되던 상황 — 미처리로 인한 고객사 이탈과 영업 실주 피드백이 누적돼, 경우의 수를 처음부터 재설계하기로 결정했습니다.
🟢 Insight
타사 식권 레퍼런스와 BGF 포스 협의에서 PG 연동 개인 추가 결제 아이디어를 도출. 토스·헥토의 API 문서를 직접 분석해 결제 구조를 설계하고, 더존 역발행 API로 국내 최초 역발행 승인 프로세스를 구현했습니다. 1만 명 이상 고객사만 커스터마이징, 나머지는 스탠다드로 정책 표준화 기준을 수립.
[ DECISION LOGIC ] 왜 결제 → UX → 정산 순서였나
판단의 근거: 우선순위를 난이도나 요청 순서가 아니라 기능 간 의존성으로 결정했습니다. 결제 구조가 정해져야 UX와 정산이 그 위에 설 수 있기 때문에, 가장 까다로운 정산을 일부러 후행으로 두었습니다.
◆ Hypothesis · 세운 가설
식권 도입 식당의 식대가 오르고, 고객사들이 일 2~3천 원을 쿠폰처럼 추가 지급하기 시작한 흐름을 포착. "임직원이 지원금을 초과해 쓰는 구간에 PG 추가 결제를 붙이면 수수료를 매출로 전환할 수 있다"는 가설을 세웠습니다.
◆ Validation · 검증한 근거
매출 상위가 아닌 방문자 수가 많은 식당의 데이터를 분석 — 결제 건수는 많은데 식권 사용액이 낮은 식당이 다수 확인됐습니다. "초과 결제 수요가 실재한다"를 데이터로 확인한 뒤 PG 연동을 결정, 출시 후 약 2개월간 결제 금액 추이를 추적해 가설을 검증했습니다.
[ DESIGN DEEP DIVE ] 가장 까다로웠던 정책

당겨쓰기 — 내일의 식권을 오늘 쓰는 기능

식권 정책 중 설계가 가장 어려웠던 건 '당겨쓰기'였습니다. 임직원이 다음 날 식권을 미리 끌어다 쓰는 기능으로, 월 사용 가능 횟수, 당겨쓴 금액의 회수 방식, 정산 시점의 처리까지 모든 경우의 수가 맞물려 있었습니다. 편의 기능 하나가 정산 구조 전체를 흔들 수 있는 만큼, 한도·횟수·회수 로직을 별도 정책으로 분리해 결제와 정산 어디서도 깨지지 않도록 설계했습니다.

02
총무타임 — B2B 스마트오피스 통합 플랫폼
침투율 95% 2024.02 – 2024.11
+
2015년산 레거시 웹으로 분산 운영되던 8개 업무 시스템을 단일 스마트오피스 앱으로 통합했습니다. 임직원·총무팀·관리자 전 사용자를 아우르는 E2E 플랫폼을 기획·설계·출시 — 사옥 이전 1개월 내 임직원 침투율 95%를 달성했습니다.
🔴 Problem
회의실·법인차량은 2015년산 웹으로 예약하고, 문제가 생기면 총무 담당자에게 직접 전화해야 했습니다. 총무팀은 하루 8시간을 수기 업무에 쏟으며 임직원 연락에 치여 정작 핵심 업무에 집중하지 못했고, 특정 부서의 회의실 독점 문제까지 복합적으로 얽혀 있었습니다.
🟡 Decision
대표의 비전인 '스마트한 사옥'을 제품으로 구현. 초기 범위에는 AI 문방구·MRO몰까지 있었지만, 이전 직후 임직원이 반드시 써야 할 핵심 8개로 범위를 좁혔습니다. 출입보안·헬프데스크를 1순위로 두되, 8개 기능을 순차가 아닌 동시 완성 목표로 — 이전 시점에 임직원이 모든 기능을 한 앱에서 쓸 수 있어야 했기 때문.
🟢 Insight
흩어진 8개 시스템을 '플랫폼 관점'에서 재설계. 핵심은 재실 감지 센서·QR 출입 등 AIoT를 업무 흐름에 자연스럽게 녹여내는 것이었습니다. 권한 체계는 사내 규정 인터뷰를 거쳐 부서관리자·총무관리자·플랫폼관리자 3단 구조로 설계해, 특정 부서의 회의실 독점 같은 이해 충돌까지 정책으로 흡수했습니다.
◆ Hypothesis · 세운 가설
"부서들이 회의 시점을 몰라 오전·오후·종일 단위로 통예약하면서 회의실이 장부상으로만 차 있다 — 실사용 기준으로 점유를 풀어주면 유휴를 회수할 수 있다"는 가설을 세웠습니다.
◆ Validation · 검증한 근거
2015년산 레거시 예약앱의 예약 내역·시간 데이터를 분석해 통예약 패턴을 확인하고, 해당 회의실을 직접 현장 확인한 결과 실제 회의가 없는 경우가 다수였습니다. "장부 점유 ≠ 실사용"을 검증한 뒤 AIoT 재실 감지 기반 자동 취소 로직을 설계.
[ DESIGN DEEP DIVE ] 예약 로직 설계

차량과 회의실은 다르게 움직인다

같은 '예약'이라도 자산 특성에 따라 로직을 다르게 설계했습니다. 법인차량은 출발·도착 거리를 필수로 기입하게 하고, 6시간 예약 후 4시간 만에 운행이 끝나면 남은 2시간을 자동으로 예약 가능 상태로 되돌립니다. 동승자 입력 기능으로 동승자도 운행일지를 쓸 수 있게 했고, 예약 시간이 지나면 임의 취소를 막았습니다. 회의실은 반대로 사용 중에도 취소를 허용하되, 예약 후 30분간 재실 센서에 감지가 없으면 자동 취소하고, 미예약 상태로 점유 중이면 안내 메시지로 다른 회의실 사용을 유도합니다. 자산을 '놀지 않게' 만드는 데 초점을 둔 설계입니다.

기획에 머물지 않고, 프로토타입을 직접 구현하며 기술적 타당성과 가설을 검증합니다. 아래는 업무 현장의 병목을 직접 코드로 해결한 결과물입니다 — 문제 정의 · 시스템 설계 · DB 설계 · 검증 · 구현까지 직접 수행했습니다.

[ 01 ] AI 기반 업무 자동화 시스템
나라장터 조달 공고 자동 분석 대시보드
React · TypeScript Flask PostgreSQL LLM API 직접 호스팅
🔴 Problem
B2B 사업이 중견 규모로 커지며 나라장터 공고 검토가 핵심 업무가 됐지만, 각 사업부 담당자가 매일 직접 공고를 확인해야 했습니다. 기존 크롤링은 제목만 가져와 상세·자격조건은 결국 사람이 다시 읽어야 했고, 공고 1건당 PDF 첨부와 면허·지역 제한까지 봐야 해 오전 4시간이 공고 검토에 소모됐습니다.
🟢 Approach
"공고를 회사 기준으로 점수화해 1차로 거르고, 가능성 높은 건만 LLM으로 정밀 분석하면 사람은 최종 판단만 하면 된다"는 가설. 룰 엔진으로 흔들림을 잡고 LLM으로 요약·리스크를 보완하는 2단 구조로 설계해, 사람을 대체하지 않고 검토 속도를 끌어올렸습니다.
